# Hyperion: Consistency Check

Hyperion's consistency check detects cross-artifact inconsistencies in programming exercises.
A programming exercise consists of four interconnected artifacts: the problem statement, the template repository, the solution repository, and the test repository.
When instructors modify one artifact without updating the others, students may encounter exercises they cannot solve — for example, the problem statement describes a method `calculateAverage` while the template declares `computeMean`.

The consistency check uses an LLM to analyze all artifacts together and dentify any inconsistencies before the exercise is published.

For the formal definition of consistency issue types and the underlying ontology, see:
[Dietrich et al., *"Automated Consistency Checking of Multi-Artifact Programming Exercises"*](https://aet.cit.tum.de/research/publications/dietrich2025koli.pdf), Koli Calling 2025.
DOI: [10.1145/3769994.3770042](https://dl.acm.org/doi/10.1145/3769994.3770042)

---

## For Instructors

Instructors with at least **Editor** role can trigger a consistency check from the exercise management view.
The report shows detected issues grouped by category and severity.
Each issue includes a description of the inconsistency, the affected artifact locations (artifact type, file path, line range), and a suggested fix.

The checker deliberately ignores pedagogical gaps — empty method stubs, `TODO` comments, and abstract method bodies in template repositories are not flagged.
Issues already covered by existing review threads on the exercise are also suppressed.

---

## Pipeline Architecture

The consistency check runs a three-phase pipeline implemented in `HyperionConsistencyCheckService`:

**Phase 1 — Language-Aware Context Filtering**

Repository files are filtered by `HyperionProgrammingLanguageContextFilterService` before any text is sent to the model.
Build artifacts, IDE configuration files, binary files, and files larger than 100 KB are stripped.
Language-specific exclusions (e.g. Gradle wrapper for Java, virtual environments for Python) are applied on top.

**Phase 2 — Structural and Semantic Analysis (concurrent)**

Two prompts execute in parallel using Spring AI's reactive `ChatClient`:
- The **structural checker** (`consistency_structural.st`) detects return-type, parameter, constructor, attribute-type, and visibility mismatches.
- The **semantic checker** (`consistency_semantic.st`) detects identifier naming inconsistencies.

Both prompts receive the filtered, line-numbered artifact text along with the exercise's programming language and any existing review threads.

**Phase 3 — Verification Pass (serial)**

The combined output of Phase 2 is passed to a dedicated verification prompt (`consistency_verification.st`).
The verifier removes false positives, deduplicates overlapping issues, corrects line ranges and descriptions, and self-verifies before returning the final list.

If the verification step fails for any reason, the pipeline falls back to the pre-verification issue list. The response is always non-null.

## Language-Aware Context Filter

`HyperionProgrammingLanguageContextFilterService` uses a **Strategy pattern** to apply language-specific file filtering before context rendering.

### Global Exclusions

Applied to every exercise regardless of language:

- Version control and IDE directories (`.git/`, `.idea/`, `.vscode/`)
- Build output directories (`bin/`, `obj/`, `out/`, `target/`, `build/`, `dist/`)
- Dependency directories (`node_modules/`, `__pycache__/`)
- Common binary extensions (`.class`, `.jar`, `.war`, `.o`, `.dll`, `.exe`, `.so`, `.png`, `.zip`, …)
- Exercise metadata files (`exercise-details.json`, `problem-statement.md`)

### Per-Language Exclusions

| Language | Additional exclusions |
|---|---|
| Java | Gradle/Maven wrappers (`gradlew*`, `mvnw*`), Eclipse project files (`.settings/`, `.classpath`, `.project`) |
| Python | Virtual environments (`venv/`, `.venv/`, `env/`), bytecode (`*.pyc`, `*.egg-info/`) |
| C | CMake build directories (`cmake-build-*/`), `CMakeCache.txt` |
| Assembler | Global exclusions only (`.o` files already covered) |
| Swift | `.swiftpm/`, `Package.resolved` |
| SQL | Global exclusions only (`.db` files already covered) |

### Safety Net and Size Guard

After exclusion patterns are applied, a **binary safety net** allows only files with extensions in a known-safe allow-list (source code, data formats, documentation) or exact filenames (`Dockerfile`, `Makefile`, `Jenkinsfile`, etc.).
Files whose content exceeds **100 KB** are dropped regardless of extension.

### Adding a New Language

To add filtering support for a new programming language:

1. Reuse `ExclusionStrategy` with a list of glob patterns (or implement `HyperionProgrammingLanguageContextFilterService.Strategy` for custom logic).
2. Call `register(...)` inside the `HyperionProgrammingLanguageContextFilterService` constructor.

```java
register(new ExclusionStrategy(ProgrammingLanguage.KOTLIN,
        List.of("glob:**/.gradle/**", "glob:**/build/**")));
```

The filter is automatically picked up the next time a consistency check runs for an exercise of that language.

## Benchmarking Workflow

This section is for **researchers and contributors** who want to evaluate a new consistency check approach against the PECV-bench dataset.

### Overview

The evaluation infrastructure connects Artemis to two external repositories:

- **[PECV-bench](https://github.com/ls1intum/PECV-bench)** — public repository containing the evaluation framework, exercise dataset (public exercises), annotation ground truth, and accumulated results from prior approaches.
- **[PECV-bench-dataset](https://github.com/ls1intum/PECV-bench-dataset)** — private repository containing confidential exercise artifacts (solution repositories, test suites) that cannot be published. Its directory structure mirrors the public `data/` subtree in PECV-bench so both are merged transparently before variant materialization.

<Callout variant={CalloutVariant.info}>
Access to PECV-bench-dataset requires explicit authorization. Contact the Hyperion maintainers to request access before running evaluations.
</Callout>

### Confidential Exercise Storage

Exercises whose solution or test artifacts cannot be made public must be stored in PECV-bench-dataset.
Use the same relative path the exercise would occupy in the public PECV-bench `data/` directory.
The benchmark scripts merge both repositories before materializing variants, so paths must match exactly.

### Dataset Versions

| Version | Exercises | Variants | Languages |
|---|---|---|---|
| V1 | 3 (ITP2425) | 91 | Java |
| V2 | 16 (ERA2021, ISE22, ITP2425, MTG26, IOS26, QCSL25) | 325 | Assembler, Java, Python, SQL, Swift |

V2 is the current evaluation standard. Use V1 for quick sanity checks or backwards-compatible comparisons.

### Prerequisites

1. A running Artemis instance (local or test server) with Hyperion enabled and an LLM configured.
2. Git access to both PECV-bench and PECV-bench-dataset.
3. Python 3.13 with pip available.
4. A configured `config.ini` (see below).

### Configuration

`supporting_scripts/hyperion/consistency-check-benchmark/config.ini` controls all benchmark parameters.
Key settings:

```ini
[Settings]
admin_user = artemis_admin
admin_password = artemis_admin
max_threads = 5
server_url = http://localhost:8080/api

[PECVCourseSettings]
pecv_bench_dir = pecv-bench
pecv_bench_dataset_dir = pecv-bench-dataset

[PECVExerciseSettings]
dataset_version = V2

[PECVConsistencyCheckSettings]
model_name = azure-openai-gpt-5-mini
model_effort = medium
consistency_check_exercises = {"V2": {"ITP2425": ["H01E01-Lectures", ...]}}
```

`model_name` and `model_effort` must match the model configured in Artemis (`spring.ai.azure.openai.chat.options.deployment-name`).
`consistency_check_exercises` controls which exercises are included in the consistency check step; it can be a subset of the full dataset.

### Running the Workflow

1. Navigate to *`consistency-check-benchmark`* folder
```shell
cd supporting_scripts/hyperion/consistency-check-benchmark
```

2. Create a virtual environment (make sure to use python 3.13):
    ```shell
    python3.13 -m venv venv
    ```

3. Activate the virtual environment:
   - On **macOS/Linux**:
     ```shell
     source venv/bin/activate
     ```
   - On **Windows**:
     ```shell
     venv\Scripts\activate
     ```

4. Install the Required Packages
```shell
pip install -r requirements.txt
```
- On **Windows**:
   ```shell
      python -m pip install --upgrade pip
      python -m pip install -r requirements.txt
   ```

   make patch.exe available (Git for Windows)

   ```shell
      setx PATH "$env:PATH;C:\Program Files\Git\usr\bin"
   ```

   restart PowerShell after this line, then continue

5. authenticate with GitHub CLI
```shell
gh auth login
```

6. run benchmark script
```shell
python3 run_pecv_bench.py
```

The script executes the following steps in sequence:

| Steps | What happens |
|---|---|
| 1–3 | Clone/pull PECV-bench and PECV-bench-dataset; install PECV-bench Python package |
| 4 | Materialize exercise variants (copy base exercise + apply per-variant patch) |
| 5–8 | Create authenticated session; create (or reuse) benchmark course on Artemis |
| 9–10 | Build variant ZIP archives; import them into Artemis via the programming-exercise import API |
| 11–12 | Fetch imported exercise IDs; trigger consistency checks concurrently |
| 13 | Generate aggregated metrics report; create a ZIP snapshot of Hyperion source files |
| 14 | Open a pull request against `artemis-approaches` in PECV-bench with results and comparison rows |

### Approach Identity

The `approach_id` is derived automatically from the current git branch and commit:

```
artemis-<branch>-<short-commit>
# e.g. artemis-feature-hyperion-verify-checker-a1b2c3d4
```

<Callout variant={CalloutVariant.warning}>
Each new approach **must** run on its own git branch. The approach ID is derived from the branch name — two implementations on the same branch produce the same ID and overwrite earlier results.
</Callout>

### Running Steps Independently

Each module can be executed standalone, which is useful when the full workflow fails partway through or when only one step needs to be repeated.
Logs produced by each script indicate exactly which step failed and which script to rerun.

| Module | Standalone entry point | When to rerun |
|---|---|---|
| `exercises.py` | `python exercises.py` | Variant materialization or ZIP creation failed |
| `course.py` | `python course.py` | Course creation or exercise ID retrieval failed |
| `consistency_check.py` | `python consistency_check.py` | One or more consistency checks failed; most common rerun case |
| `report.py` | `python report.py` | Report generation failed after checks completed |
| `code_snapshot.py` | `python code_snapshot.py` | Source snapshot ZIP was not created |
| `merge_request.py` | `python merge_request.py` | Pull request creation failed |

<Callout variant={CalloutVariant.info}>
Steps 11–12 (`consistency_check.py`) require an active session and an existing course. When running it standalone, the script automatically re-authenticates and retrieves the course ID before triggering checks.
</Callout>

When running `consistency_check.py` standalone, note the `approach_id` printed at the end and set it manually in `report.py`, `code_snapshot.py`, and `merge_request.py` before running those scripts.

## Known Limitations

**LLM non-determinism.** Results vary between runs. For reliable metric comparisons, run each approach at least three times and average the results.

**Course deletion may not complete on the first attempt** due to network latency or server load. If the script fails at the course deletion step, re-run it — the step is idempotent.

**Exercise import may fail if the local VCS already contains exercises with the same name.** This happens when a previous benchmark run left stale data in the Artemis working directory. To fix it, remove all folders inside `repos/` and `local-vcs-repos/` in the Artemis project root, then re-run the script.
