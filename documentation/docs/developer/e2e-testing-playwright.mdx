# E2E Testing with Playwright

## Background

The Playwright test suite contains system tests that verify the most important user-facing workflows in Artemis.
System tests exercise the complete technology stack — frontend, backend, database, and integrated services — by simulating real browser interactions.
In order to detect regressions before they reach the `develop` branch, the pipeline runs the Playwright suite on every pull request and on every push to `develop`.

Artemis uses GitHub Actions as its CI platform. E2E tests run on self-hosted runners rather than GitHub-hosted runners because
they require substantial compute resources, access to a PostgreSQL database, and a locally deployed Artemis instance.
Self-hosted runners do not start from a clean state between jobs; the pipeline cleans up leftover artifacts explicitly between phases.

To keep feedback cycles short, the pipeline uses a **two-phase execution model**: Phase 1 runs only the tests
mapped to the files changed in the pull request, and Phase 2 runs the remaining tests only when Phase 1 passes.
This gives developers targeted feedback within a few minutes for the tests most likely to surface a regression,
without waiting for the full suite to complete.

## Running E2E Tests Locally

E2E tests require the full Artemis stack to be running: database, server, and client.
The **recommended approach is the [fast local runner](#fast-local-runner-recommended-for-development)**,
a single script that handles all of this automatically.
Alternatively, you can use the Docker-based supporting scripts or set everything up manually.

### Fast local runner (recommended for development)

Use `run-e2e-tests-local-fast.sh` in the repository root for the fastest local development experience.
Unlike the Docker-based approach, this script runs only PostgreSQL in Docker while running the server and client directly on the host.
Services stay running between test runs, so re-runs only take seconds.

The script automatically kills any processes on ports 8080 (server), 9000 (client), and 7921 (local VC SSH)
before starting, so you do not need to stop conflicting processes manually.

**Prerequisites:** Docker, Java 25+, Node.js 24+, npm.

**Architecture:**

- **Docker:** PostgreSQL only (port 5432)
- **Host:** Spring Boot server via `./gradlew bootRun -x webapp` (port 8080)
- **Host:** Angular client via `npm start` (port 9000, proxies API calls to 8080)
- **Host:** Playwright connects to `http://localhost:9000`

**First run (starts everything):**

```bash
./run-e2e-tests-local-fast.sh
```

This starts PostgreSQL, boots the server (~30–90 s), starts the client (~10–20 s), then runs all Playwright tests.

**Run specific tests:**

Use `--filter` with a pattern (supports regex) to run only matching tests:

```bash
# Run tests with "Quiz" in the name
./run-e2e-tests-local-fast.sh --filter "Quiz"

# Run tests matching multiple patterns
./run-e2e-tests-local-fast.sh --filter "ExamAssessment|SystemHealth"
```

**Re-run tests (services already running):**

After the first run, services stay running. Skip startup for faster re-runs:

```bash
# Run all tests (skips starting services that are already running)
./run-e2e-tests-local-fast.sh --skip-server --skip-client --skip-db

# Run only specific tests
./run-e2e-tests-local-fast.sh --skip-server --skip-client --skip-db --filter "Quiz"

# Open Playwright UI for interactive debugging
./run-e2e-tests-local-fast.sh --skip-server --skip-client --skip-db --ui
```

**Stop everything:**

```bash
./run-e2e-tests-local-fast.sh --stop
```

**Available options:**

| Flag | Description |
|------|-------------|
| `--stop` | Kill server, client, and database; exit |
| `--filter <pattern>` | Run only tests matching the pattern (supports regex, e.g., `"Quiz"` or `"Quiz\|Exam"`) |
| `--skip-server` | Reuse already-running server |
| `--skip-client` | Reuse already-running client |
| `--skip-db` | Reuse already-running PostgreSQL |
| `--headed` | Run Playwright in headed mode (visible browser) |
| `--ui` | Open Playwright UI mode for interactive debugging |
| `--video` | Enable video recording (off by default to save CPU) |
| `--coverage` | Enable JS coverage collection via monocart (off by default; requires extra memory) |
| `--debug` | Show server and client output inline (normally written to log files only) |

Logs and PID files are stored in the `.e2e-local/` directory (gitignored).

### Supporting scripts overview

The following helper scripts live in `supporting_scripts/playwright` and provide an alternative Docker-based workflow:

- `runArtemisInDocker_macOS.sh` / `runArtemisInDocker_linux.sh`: start the Artemis server, client, and database in Docker containers.
- `setupUsers.sh`: create the Playwright test users in a running Artemis instance.
- `startPlaywright.sh`: run the full Playwright suite headless in the terminal.
- `startPlaywrightUI.sh`: launch Playwright in UI mode for interactive debugging.
- `prepareVSCodeForE2ETests.sh`: install dependencies and patch the Playwright config so VS Code can discover and run tests from the Testing sidebar.
- `checkFlakiness.sh`: run a specific test repeatedly to detect non-deterministic failures before pushing to CI.

The three steps for the Docker-based approach:

1. Start Artemis with `runArtemisInDocker_macOS.sh` or `runArtemisInDocker_linux.sh`. Keep the process running and open a separate shell.
2. Create test users with `setupUsers.sh` (skip if users already exist).
3. Run tests headless with `startPlaywright.sh`, interactively with `startPlaywrightUI.sh`, or from VS Code after running `prepareVSCodeForE2ETests.sh`.

### Manual setup

Use manual setup when you need IntelliJ debugger integration or want full control over the running Artemis instance.
Follow the [server setup guide](./setup.mdx) to prepare a locally running Artemis instance first.

#### 1. Install dependencies

```bash
cd src/test/playwright
npm install
```

#### 2. Customize Playwright configuration

All configuration is stored in `playwright.env`. The default configuration for a local ICL setup:

```text
PLAYWRIGHT_USERNAME_TEMPLATE=artemis_test_user_
PLAYWRIGHT_PASSWORD_TEMPLATE=artemis_test_user_
ADMIN_USERNAME=artemis_admin
ADMIN_PASSWORD=artemis_admin
ALLOW_GROUP_CUSTOMIZATION=true
STUDENT_GROUP_NAME=students
TUTOR_GROUP_NAME=tutors
EDITOR_GROUP_NAME=editors
INSTRUCTOR_GROUP_NAME=instructors
BASE_URL=http://localhost:9000
EXERCISE_REPO_DIRECTORY=test-exercise-repos
FAST_TEST_TIMEOUT_SECONDS=45
SLOW_TEST_TIMEOUT_SECONDS=90
```

Make sure `BASE_URL` matches your Artemis client URL and the admin credentials match your local Artemis admin user.

#### 3. Configure test users

Generate test users with the script in `supporting_scripts/playwright`:

```bash
setupUsers.sh
```

You can review user IDs and their roles in `src/test/playwright/support/users.ts`.
Usernames are formed by appending the user ID to `PLAYWRIGHT_USERNAME_TEMPLATE`.

#### 4. Set up Playwright and browser binaries

```bash
npm run playwright:setup-local
npm run playwright:init
```

#### 5. Run tests

Open the graphical UI:

```bash
npm run playwright:open
```

Or run all tests in the terminal:

```bash
npm run playwright:test
```

To run a specific file:

```bash
npx playwright test src/test/playwright/e2e/atlas/CompetencyManagement.spec.ts
```

To run a specific test or suite within a file, use the `-g` flag:

```bash
npx playwright test src/test/playwright/e2e/course/CourseManagement.spec.ts -g "Course creation"
```

## Test Organization

### Test projects: fast and slow tests

Playwright is configured with two test projects that run concurrently:

| Project | Tag | Timeout (local) | Timeout (CI) | Use for |
|---------|-----|-----------------|--------------|---------|
| `fast-tests` | `@fast` or no tag | 45 s | 60 s | Tests that complete quickly and do not involve long-running background jobs |
| `slow-tests` | `@slow` | 90 s | 90 s | Tests that wait for CI/CD builds, large data processing, or other asynchronous operations |

Tag your test suite with `{ tag: '@slow' }` when it involves build pipelines or polling for background jobs; otherwise omit the tag (defaults to `fast-tests`):

```typescript
// Fast test (default)
test.describe('Competency Management', { tag: '@fast' }, () => { ... });

// Slow test
test.describe('Programming Exercise Participation', { tag: '@slow' }, () => { ... });
```

By default, tests run with **5 parallel workers**. Each worker controls a separate browser instance.
The number of workers is configurable via the `TEST_WORKER_PROCESSES` environment variable.

<Callout variant={CalloutVariant.warning}>
    <p>Using more worker processes divides available compute resources among workers. Too many workers cause resource
    contention, longer individual test execution, and increased timeout risk. Adjust <code>TEST_WORKER_PROCESSES</code>
    to match your machine's capacity.</p>
</Callout>

Tests are configured with **2 automatic retries** on failure and a **10 s expect timeout** for assertions.
Traces are collected on the first retry, and videos are recorded for every test run to aid debugging.

### Global setup

Before any test runs, `init/global-setup.ts` executes once. It provisions the Artemis instance with seed data
needed by certain tests (e.g., persistent seed courses used by Atlas tests) and verifies that the application
is reachable. If global setup fails, no tests are executed.

## Writing E2E Tests

### Understanding the system and requirements

Before writing tests, develop a thorough understanding of the workflow you are covering. Consult the original developer
or a person actively working on the component to understand its intended behavior across all user roles.

### Identify main and edge scenarios

Identify the primary ways the component is used and test them with all relevant user roles and representative inputs.
Also cover edge cases: inputs that are not supposed to work (e.g., overly long text), and actions that should produce
error states.

### Write tests as development progresses

Write tests alongside feature development rather than after the fact. This keeps code testable and makes issues
easier to identify and fix early.

### Keep tests focused and independent

Each test should verify one specific behavior. Tests must not depend on each other or on external factors like the
current date. Use API calls for setup tasks (e.g., creating a course) and UI interaction for the behavior under test.
Clean up after each test in `afterEach`.

### Use descriptive test names

Test names appear in CI comments and failure reports. Write them so that a reader who has not seen the code
understands what scenario failed.

### Do not ignore failing tests

Investigate consistently failing tests immediately. A test that always fails is either broken or exposing a real defect.
Leaving it broken degrades trust in the entire suite.

### Regularly review and refactor tests

Tests accumulate technical debt just like production code. Review tests periodically for unnecessary complexity,
duplicated setup logic, or outdated assumptions.

## Playwright-Specific Best Practices

### 1. Use page objects for common interactions

Page objects encapsulate the details of a page's structure and interactions. They are stored in
`src/test/playwright/support/pageobjects/`. Each page object is a class containing a Playwright `page` instance
and methods for common user actions. Page objects are registered as fixtures for easy injection into tests.

### 2. Use fixtures

Fixtures manage test initialization and teardown so each test starts from a known state.
All page objects and common commands (e.g., `login`) are available as fixtures defined in `support/fixtures.ts`.

To create a fixture:

```typescript
export type ArtemisPageObjects = {
    loginPage: LoginPage;
}

export const test = base.extend<ArtemisPageObjects>({
    loginPage: async ({ page }, use) => {
        await use(new LoginPage(page));
    }
});
```

Inject the fixture in a test:

```typescript
test('Test name', async ({ loginPage }) => {
    // use loginPage
});
```

### 3. Use `data-testid` locators

Prefer `data-testid` attributes over CSS selectors, text content, or ARIA roles as the primary locator strategy.
Text-based and role-based locators break whenever the display text or component structure changes,
turning a UI rename into a test failure that has nothing to do with a regression.
A `data-testid` attribute survives UI refactors as long as the attribute itself is preserved.

If a component does not have a `data-testid`, add one to its template:

```html
<button data-testid="submit-competency-button">Save</button>
```

Then reference it in the test:

```typescript
await page.getByTestId('submit-competency-button').click();
```

Avoid `nth()` and `:nth-child` selectors; they couple tests to DOM position and break when sibling elements are added.

### 4. Consider element actionability

Playwright automatically waits for elements to be visible and enabled before performing actions such as `click()` or `fill()`.
For cases where you need to access inner text or other properties before interaction, use `waitFor()` on the locator:

```typescript
await page.locator('.clone-url').waitFor({ state: 'visible' });
const urlText = await page.locator('.clone-url').innerText();
```

<Callout variant={CalloutVariant.warning}>
    <p>Do not use <code>page.waitForSelector()</code>. It only checks DOM presence, not actionability.
    Always prefer <code>locator.waitFor()</code>.</p>
</Callout>

Use `waitForLoadState()` only when you need the entire page to finish loading, not when waiting for a specific element:

```typescript
await page.waitForLoadState('load');
```

<Callout variant={CalloutVariant.warning}>
    <p>Avoid waiting for load state when you are interested in a specific element — use <code>locator.waitFor()</code> instead.</p>
</Callout>

### 5. AI-assisted test authoring with the Playwright MCP server

The repository integrates a Playwright Model Context Protocol (MCP) server that exposes the live application DOM
to AI coding assistants such as GitHub Copilot and Claude Code.

Without the MCP server, AI assistants can generate test structure and boilerplate but produce incorrect locators
because they have no access to the running page state.
With the MCP server active, the assistant can inspect the current DOM, retrieve the `data-testid` attributes
attached to elements, and generate locators that reference those IDs directly.

To use it, start Artemis locally (via the fast runner or any other method) and enable the Playwright MCP
integration in your editor's AI assistant settings. The assistant can then:
- Inspect the current page structure and available `data-testid` attributes
- Generate accurate locators for the elements involved in the workflow
- Produce a first-draft test that closely matches the actual DOM

Review the generated test by running it locally, confirm that the interaction sequence matches the intended workflow,
and correct any steps where the assistant misread the application state.
This approach significantly reduces the gap between a first draft and a passing test.

## CI Pipeline

### Overview: two-phase execution

Every pull request triggers a two-phase E2E pipeline. Phase 1 runs only the tests mapped to the files changed in
the pull request; Phase 2 runs all remaining tests, but only when Phase 1 passes.
This architecture ensures that developers receive targeted feedback within minutes for the tests most likely to surface
a regression, while still running the full suite before merge.

<Image src={e2eCiPipelineSsd} alt="Two-phase CI pipeline activity diagram showing Developer, CI Pipeline, Helios, and Reports Dashboard swimlanes" size={ImageSize.large} caption="Two-phase CI pipeline activity diagram. Phase 1 runs change-relevant tests first. If Phase 1 passes, Phase 2 runs the remaining tests. Both phases query Helios for flakiness scores and upload artifacts to the Reports Dashboard before posting the PR comment." />

**Observed performance** (across 89 measured runs):

| Metric | Phase 1 | Phase 2 | Full suite (single-phase baseline) |
|--------|--------:|--------:|----------------------------------:|
| Median duration | 4.3 min | 23.8 min | 27.7 min |
| P90 duration | 8.3 min | 29.4 min | 32.2 min |
| Pass rate | 82% | 64.4% (of runs that reach it) | — |

When Phase 1 detects a failure, developers receive feedback within approximately four minutes and Phase 2
does not consume compute resources.

**For pushes to `develop` or `main`:** the pipeline skips the two-phase logic and runs the full suite in a single job.

### Selective test execution and `e2e-test-mapping.json`

The `determine-tests` CI job reads `.ci/E2E-tests/e2e-test-mapping.json` to decide which tests to include in Phase 1.
The mapping file lists, for each Artemis module, the source paths that belong to it and the test paths that cover it:

```json
{
  "mappings": {
    "atlas": {
      "sourcePaths": [
        "src/main/java/de/tum/cit/aet/artemis/atlas/",
        "src/main/webapp/app/atlas/"
      ],
      "testPaths": ["e2e/atlas/"]
    },
    "exam": {
      "sourcePaths": [
        "src/main/java/de/tum/cit/aet/artemis/exam/",
        "src/main/webapp/app/exam/"
      ],
      "testPaths": ["e2e/exam/"]
    }
  },
  "alwaysRunTests": [
    "e2e/Login.spec.ts",
    "e2e/Logout.spec.ts",
    "e2e/SystemHealth.spec.ts"
  ],
  "runAllTestsPatterns": [
    "src/main/resources/config/",
    "docker/",
    "build.gradle",
    "angular.json"
  ]
}
```

The `determine-tests` job compares the PR branch against the base branch and maps each changed file to its module.
The union of the test paths from all matched modules, plus the `alwaysRunTests` entries, forms the Phase 1 set.
The remaining entries from `allTestPaths` form the Phase 2 set.

**Special cases:**
- If a changed file matches a `runAllTestsPatterns` entry (e.g., a Docker or Gradle file), the pipeline skips the two-phase split and runs all tests in a single job.
- If only Playwright spec files were changed (no source changes), Phase 2 is skipped and only Phase 1 runs.
- If no module mapping is found for a changed file, it falls through to Phase 2.

**Adding tests for a new module:** add an entry to the `mappings` object in `e2e-test-mapping.json` and add the test path(s) to `allTestPaths`.
This ensures Phase 1 selects the new tests when the corresponding source files change, and Phase 2 covers them otherwise.

### PR comment reporting

The pipeline posts a single comment on each pull request and updates it progressively as phases complete.
The comment is never re-created; it accumulates results in place.

A typical comment after both phases shows:

- A phase table with status (✅ / ❌), test count, passed, skipped, failed, and wall-clock time for each phase
- The test file paths assigned to each phase
- Links to the workflow run and the full HTML test report on the Reports Dashboard
- A collapsible section listing each failed test name and its duration
- A flakiness score table for any failed tests (when Helios data is available)
- An overall result line summarizing the outcome in plain language

Phase 1 results are posted immediately when Phase 1 finishes. If Phase 2 runs, the same comment is updated with
Phase 2 results and the final overall status.

See [Find example PR comments](#example-pr-comments) for real examples of each possible outcome.

### Helios flakiness scores

After each phase completes, the pipeline queries the Helios API to retrieve flakiness scores for any tests that failed.
The scores are included in the PR comment alongside the failed test names.

A **flakiness score** reflects how often a test has non-deterministically produced different results over recent CI runs.
A high score means the test has a history of failing without a code change and the current failure may be a flake.
A low score means the test is normally stable and the failure is more likely a genuine regression.

The Helios integration is active for all PR runs. Scores appear as a table in the `❌ Failed Tests` section of the
PR comment when Helios returns data for the failed tests.

### Reports Dashboard

The Reports Dashboard is a web interface that stores and serves all E2E test artifacts from CI runs.
It is accessible from the link in the PR comment.

The dashboard provides:

- **Aggregate view:** total runs, 30-day pass rate, average flakiness rate, pass rate trend chart, and runtime-by-phase chart
- **Per-run view:** phase breakdown, test-level results with status and duration, links to the full HTML report and coverage report
- **Failure detail view:** the full assertion error, stack trace, and an embedded video recording of the browser at the moment of failure

The dashboard indexes runs by pull request and commit. Artifacts are retained until the pull request is merged.

### Example PR comments

Three outcomes are possible for a two-phase run:

1. **Both phases pass** — The comment shows two green rows in the phase table and an overall ✅ result. See [PR #12516](https://github.com/ls1intum/Artemis/pull/12516) for an example.

2. **Phase 1 passes, Phase 2 fails** — The comment shows a green Phase 1 row and a red Phase 2 row. The failed test section lists Phase 2 failures. See [PR #12489](https://github.com/ls1intum/Artemis/pull/12489) for an example.

3. **Phase 1 fails, Phase 2 skipped** — The comment shows a red Phase 1 row and no Phase 2 row. Developers receive targeted failure feedback within a few minutes without waiting for the rest of the suite. See [PR #12487](https://github.com/ls1intum/Artemis/pull/12487) for an example.

## CI Deployment

Every CI E2E run requires a complete Artemis deployment on the self-hosted runner.
The runner starts all required services using Docker Compose and executes Playwright tests on the host.

<Image src={e2eHwSwMapping} alt="Hardware-software mapping showing five nodes: Developer Machine, GitHub, Self-Hosted CI Runner, Reports Dashboard, and Helios" size={ImageSize.large} caption="Hardware-software mapping of the E2E testing infrastructure. The Developer Machine pushes to GitHub, which triggers the Actions Orchestrator. The Self-Hosted CI Runner executes tests and sends results to the Reports Dashboard and PR comments via the Report and Communication Handler. The Helios node provides flakiness scores." />

The system operates across five nodes:

### Developer Machine

The developer writes and commits test code in the Artemis repository on their local machine,
then pushes changes to GitHub, which triggers the CI pipeline.

### GitHub

GitHub hosts two components. The **Actions Orchestrator** receives push and pull request events and
schedules pipeline jobs on the self-hosted runner. The **PR System** exposes the API through which
the pipeline posts and updates pull request comments.

### Self-Hosted CI Runner

The runner is the primary execution node. It carries two responsibilities.

The **Runner** component starts the application stack and executes the Playwright test suite.
The stack consists of:

- **PostgreSQL** — started via `docker compose` (`docker/postgres.yml`), exposed on port 5432
- **Artemis application** — the Docker image built from `docker/artemis/Dockerfile`,
  configured via files in `docker/artemis/config/` and environment variables from GitHub secrets
- **Playwright** — executed directly on the runner host (not in Docker), connecting to the Artemis instance at `http://localhost:9000`

The **Report and Communication Handler** aggregates run outputs, uploads artifacts to the Reports Dashboard,
retrieves the resulting run permalink, queries the Helios API for flakiness scores of failed tests,
and posts the consolidated comment to the GitHub PR System.

Self-hosted runners do not start from a clean state between jobs. The pipeline explicitly cleans up
test result files from Phase 1 before Phase 2 begins to prevent workspace pollution from inflating test counts.

### Reports Dashboard

The dashboard stores test artifacts and serves them via a web interface backed by a SQLite database.
When the pipeline uploads a run, the dashboard generates a permalink to that specific run entry, which
the Report and Communication Handler includes in the PR comment.

### Helios

The Helios node provides the flakiness scoring service. The Report and Communication Handler sends
the names of any failed tests to the Helios API after each phase and receives scores reflecting
each test's historical non-determinism rate.

## Maintenance

### Updating the Playwright Docker image

The Playwright Docker image used in CI (`docker/playwright.yml`) pins a specific version that includes
particular browser binaries. Update this image monthly to keep browsers current.
Check the [Playwright release notes](https://playwright.dev/docs/release-notes) and update the image tag accordingly.

### Updating `e2e-test-mapping.json`

When a new Artemis module is created or an existing module is significantly reorganized:

1. Add a new entry to `mappings` in `.ci/E2E-tests/e2e-test-mapping.json`.
2. List the relevant `sourcePaths` (Java and Angular directories for the module).
3. List the corresponding `testPaths` (spec file paths or directory paths).
4. Add all new test paths to `allTestPaths` so Phase 2 covers them on unrelated PRs.

Keeping the mapping accurate ensures Phase 1 selects the right tests when changes are made to each module.

### Coverage reports

The pipeline collects JavaScript coverage via `monocart-reporter` and uploads the merged report as an artifact.
Coverage is off by default when running locally (to save memory) and enabled with `--coverage` in the fast runner.
In CI, monocart merges coverage from parallel and sequential phases and outputs LCOV and HTML reports.
