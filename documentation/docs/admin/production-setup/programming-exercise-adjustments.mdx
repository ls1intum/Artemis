# Programming Exercise Adjustments

There are several variables that can be configured when using programming exercises.
They are presented in this separate section to keep the 'normal' setup guide shorter.


## Path Variables

There are variables for several paths:

- `artemis.repo-clone-path`

  Repositories that the Artemis server needs are stored in this folder.
  This e.g. affects repositories from students which use the online code editor or
  the template/solution repositories of new exercises, as they are pushed to the VCS after modification.

  Files in this directory are usually not critical, as the latest pushed version of these repositories are
  also stored at the VCS.
  However, changes that are saved in the online code editor but not yet committed will be lost when
  this folder is deleted.

- `artemis.repo-download-clone-path`

  Repositories that were downloaded from Artemis are stored in this directory.

  Files in this directory can be removed without loss of data, if the downloaded repositories are still present
  at the VCS.
  No changes to the data in the VCS are stored in this directory (or they can be retrieved by performing
  the download-action again).

- `artemis.template-path`

  Templates are available within Artemis.
  The templates should fit to most environments, but there might be cases where one wants to change the templates.

  This value specifies the path to the templates which should overwrite the default ones.
  Note that this is the path to the folder where the `templates` folder is located, not the path to the
  `templates` folder itself.



## Templates

Templates are shipped with Artemis (they can be found within the `src/main/resources/templates` folder in GitHub).
These templates should fit well for many deployments, but one might want to change some of them for special deployments.

As of now, you can overwrite the `jenkins` folders that is present within the `src/main/resources/templates` folder
by placing a `templates/` directory with the same structure next to the Artemis `.war` archive.
Files that are present in the file system will be used, if a file is not present in the file system,
it is loaded from the classpath (e.g. the `.war` archive).

We plan to make other folders configurable as well, but this is not supported yet.


### Jenkins Template

The build process in Jenkins is stored in a `config.xml`-file (in `src/main/resources/templates/jenkins/`).
It is extended by a `Jenkinsfile` in the same directory that will be placed inside the `config.xml` file.
The `Jenkinsfile` handles the functionality shared by all programming languages like checking out the repositories and
loading the actual exercise-specific pipeline script from the Artemis server.

<Callout variant={CalloutVariant.info}>
    When overriding the `Jenkinsfile` with a custom one, note that it **must** start either

    - with `pipeline` (there must not be a comment before pipeline, but there can be one at any other position,
      if the Jenkinsfile-syntax allows it)
    - or the special comment `// ARTEMIS: JenkinsPipeline` in the first line.
</Callout>

The actual programming language or exercise-type specific pipeline steps are defined in the form of
[scripted pipelines](https://www.jenkins.io/doc/book/pipeline/syntax/#scripted-pipeline).
In principle, this is a Groovy script which allows structuring the pipeline into smaller methods and allows
conditionally executing steps, but inside still allows the core structure blocks from
[declarative pipelines](https://www.jenkins.io/doc/book/pipeline/syntax/#declarative-pipeline).
You can override those `pipeline.groovy` files with the template mechanism described above.

Inside the `pipeline.groovy` some placeholders exist that will be filled by Artemis upon exercise creation from the
server or exercise settings:

| Variable                       | Replacement                                                    | Origin                 |
|--------------------------------|----------------------------------------------------------------|------------------------|
| `#dockerImage`                 | The container image that the tests will run in.                | Server configuration   |
| `#dockerArgs`                  | Additional flags passed to Docker when starting the container. | Server configuration   |
| `#isStaticCodeAnalysisEnabled` | Defines if static code analysis should be performed.           | Exercise configuration |

The `pipeline.groovy` file can be customized further by instructors after creating the exercise from within
Artemis via the ‘Edit Build Plan’ button on the details page of the exercise.


## Caching Maven Dependencies with Sonatype Nexus {/* #caching-sonatype-nexus */}

With Sonatype Nexus you can run a caching server in your local network for Maven dependencies.
An alternative approach for caching is with Docker volumes, see [this section](#caching-docker-volumes).

<Callout variant={CalloutVariant.info}>
  The following steps assume `artemis.example.com` is the host `10.0.73.42` and is using port `8443` for the cache.
  Adapt the URLs for your actual setup.
</Callout>


### Sonatype Nexus Setup

1. Set up Sonatype Nexus to run on `artemis.example.com:8443`, e.g. in a [Docker container](https://hub.docker.com/r/sonatype/nexus3/) behind a [proxy](https://help.sonatype.com/en/run-behind-a-reverse-proxy.html).
2. In the initial setup steps: Allow anonymous access.
3. Set up the Maven proxy repository:
    a. Create a new repository (*Repository - Repositories - Create repository*) of type `maven2 (proxy)` with name `maven-proxy`.
    b. The remote URL is https://repo1.maven.org/maven2/.
4. Optionally create a new cleanup policy under *Repository - Cleanup Policies*
    1. Format: `maven2`
    2. Release type: Releases & Pre-releases/Snapshots
    3. Cleanup criteria: e.g. ‘Component Usage 14’ will remove all files that have not been downloaded for 14 days.
    4. You can now add this cleanup policy to the policies in the repository you created earlier.


### Adding proxy to a Maven build

The following changes have to be made inside the `tests` repository.

#### Option 1

Configure Maven so that it can find your Maven cache:

```xml title="pom.xml"
<repositories>
    <repository>
        <id>artemis-cache</id>
        <url>https://artemis.example.com:8443/repository/maven-proxy/</url>
    </repository>
</repositories>
<pluginRepositories>
    <pluginRepository>
        <id>artemis-cache</id>
        <url>https://artemis.example.com:8443/repository/maven-proxy/</url>
    </pluginRepository>
</pluginRepositories>
```


#### Option 2 (more rigorous alternative)

This setup forces Maven to exclusively download dependencies from the own proxy.

```xml title=".mvn/local-settings.xml"
<settings xmlns="http://maven.apache.org/SETTINGS/1.2.0"
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.2.0 https://maven.apache.org/xsd/settings-1.2.0.xsd">
<mirrors>
    <mirror>
    <id>artemis-cache</id>
    <name>Artemis Cache</name>
    <url>https://artemis.example.com:8443/repository/maven-proxy/</url>
    <mirrorOf>*</mirrorOf>
    <blocked>false</blocked>
    </mirror>
</mirrors>
  </settings>
```


```bash title=".mvn/maven.config"
--settings
./.mvn/local-settings.xml
```


#### Adding proxy to a Gradle build

The following changes have to be made inside the `tests` repository.

```groovy title="build.gradle"
repositories {
    maven {
        url "https://artemis.example.com:8443/repository/maven-proxy/"
    }
    // …
}
```


```kotlin title="build.gradle.kts"
repositories {
    maven {
        url = uri("https://artemis.example.com:8443/repository/maven-proxy/")
    }
    // …
}
```


## Caching via Per-Agent Bind Mounts {/* #caching-bind-mounts */}

For self-hosted Integrated Code Lifecycle setups that hit Maven Central rate limits (HTTP 429) but do not want to
operate a separate Sonatype Nexus instance, the build agent can bind-mount persistent Maven and Gradle dependency
caches into every build container. Dependencies are then downloaded once per agent (cold miss) instead of once per
submission, which typically drops outbound traffic to Maven Central by ~99% in steady state.

This option is complementary to, not a replacement for, the [Sonatype Nexus approach](#caching-sonatype-nexus) — both
can coexist if needed.

### Configuration

Set the following keys in `application-buildagent.yml` on each build agent:

```yaml
artemis:
    continuous-integration:
        build-container-cache:
            maven: /var/cache/artemis-buildagent/m2
            gradle: /var/cache/artemis-buildagent/gradle
            read-only: false                # set to true to mount both caches read-only (see below)

            # --- Cache cleanup (drain-and-prune, JVM-driven) ----------------------------------
            cleanup-enabled: true                              # default; flip to false to disable cleanup entirely
            cleanup-cron: "0 0 5 * * *"                        # Spring 6-field cron; default daily 05:00 UTC
            cleanup-max-age-days: 30                           # phase 1: delete files unread for N days
            maven-max-size: 3GB                                # phase 2: hard cap on the Maven cache total size
            gradle-max-size: 6GB                               # phase 2: hard cap on the Gradle cache total size
            size-low-watermark-ratio: 0.75                     # evict LRU until total drops below this × max-size
```

The bind-mount keys (`maven`, `gradle`, `read-only`) are the minimal subset required to enable caching; the
`cleanup-*` keys are optional and default to the values shown above. The cleanup mechanism is described in detail
in the [Cleanup section](#caching-bind-mounts-cleanup).

> **Size unit convention.** Spring's `DataSize` parser treats `3GB` as **3 GiB** (3 × 1024³ ≈ 3.22 × 10⁹ bytes),
> not 3 × 10⁹ bytes. This matches `df -h` / `du -h` output. If you prefer decimal multiples (3 × 10⁹), use the
> explicit byte count: `maven-max-size: 3000000000`. The 7 % difference rarely matters in practice but is worth
> knowing when sizing against a tight disk.

Either path key can be set independently. When set, the corresponding host directory is bind-mounted into every
build container at `/root/.m2` or `/root/.gradle` respectively, which is `$HOME/.m2` / `$HOME/.gradle` for the
default root-based Java build images shipped with Artemis (`ls1tum/artemis-maven-template`, `ls1tum/artemis-gradle-template`,
etc.). When unset (the default), no bind mounts are added and the behaviour is unchanged.

> **Image compatibility:** the container-side mount points are hard-coded to `/root/.m2` and `/root/.gradle`. If you
> override the default Artemis build image with one whose `$HOME` is not `/root` (e.g. the upstream Docker Hub
> `gradle` image runs as user `gradle` with `$HOME=/home/gradle`), the bind mount lands somewhere the build tool
> does not consult — the build succeeds but the cache is silently ineffective. Either keep the Artemis-shipped
> images, build a derivative image that sets `$HOME=/root`, or open a follow-up for per-image mount paths.

The optional `read-only` flag (default `false`) controls the bind access mode for both caches at once:

- `read-only: false` — the build container can write to the cache. The first submission warms it; subsequent
  submissions on the same agent reuse the entries. This is the fastest opt-in path but a security trade-off
  applies (see the cache-poisoning caveat below).
- `read-only: true` — both caches are mounted with `:ro`. Untrusted student submissions cannot mutate the shared
  cache, so cache poisoning is structurally impossible. **The cache must be fully populated before the flag is
  enabled, otherwise builds will fail at dependency resolution.** The read-only bind shadows whatever was in the
  image's `~/.m2` / `~/.gradle` layer (including the dependencies baked into `ls1tum/artemis-maven-template`), and
  Maven / Gradle cannot write fetched artifacts into a read-only local repository — a missing transitive
  dependency therefore aborts the build instead of being re-downloaded.
  Populate the cache out of band, for example with a trusted warm-up that runs once per agent with the same
  image the student builds use:

  ```bash
  # Maven (mount the seed pom and the cache, override the image entrypoint so we run mvn directly)
  docker run --rm --entrypoint mvn \
      -v /var/cache/artemis-buildagent/m2:/root/.m2 \
      -v "$PWD/seed-pom.xml":/seed/pom.xml \
      ls1tum/artemis-maven-template:java17-25 \
      -f /seed/pom.xml dependency:resolve

  # Gradle (uses a Gradle build that resolves the same dependency set)
  docker run --rm \
      -v /var/cache/artemis-buildagent/gradle:/root/.gradle \
      -v "$PWD/seed-project":/seed -w /seed \
      ls1tum/artemis-gradle-template:java17-25 \
      gradle --no-daemon dependencies
  ```

  Re-run the warm-up whenever the dependency set the exercises declare changes. As an alternative, flip the
  flag back to `false` long enough to let normal submissions repopulate, then flip it back.

### Host directory requirements

- The directories must exist before the build agent starts; the agent does not create them.
- They must be writable by the build-container user. The recommended permissions depend on the build image:
  - **Root-based images (Artemis defaults):** create the directories owned by `root:root` with mode `0755`. The
    container's root user can read and write; no other host user can.
  - **Custom non-root images:** `chown` the host directory to the UID/GID the container runs as, or set a POSIX
    ACL granting that UID write access. Avoid world-writable permissions when a more precise grant works.
  - **Fallback only:** mode `0777` (world-writable) is the easiest last-resort workaround when an agent runs a mix
    of root and non-root build images, but be aware that it lets any local process on the build-agent host modify
    the cache — only acceptable on machines that run nothing besides the Artemis build agent.
- The cache grows over time and is pruned by a JVM-driven daily cleanup; see the
  [Cleanup section](#caching-bind-mounts-cleanup) below.
- Pair the cleanup with a disk-space alert on the cache filesystem. The exact mechanism depends on your monitoring
  stack — for example, a Prometheus + node_exporter setup can fire when
  `(node_filesystem_avail_bytes / node_filesystem_size_bytes) < 0.2` for the mount point, which leaves enough head
  room to investigate before builds start failing. The cache is recoverable (it is just a download mirror), so
  alerting at 80% used is plenty.

### Cleanup (drain-and-prune) {/* #caching-bind-mounts-cleanup */}

The cache is pruned by a scheduled task that runs **inside the build agent JVM**, not by a host-level cron job.
The agent first pauses itself, waits for in-flight builds to finish, runs the prune, and then resumes. Driving the
prune from inside the agent avoids two classes of bug that an out-of-band `find -delete` cron is exposed to:

1. **Race-free deletion.** A host cron would race with running builds reading the cache — deleting a file that an
   active Maven/Gradle process is about to open would force the build to re-download from Maven Central. The agent
   pauses first, so no build is reading the cache during the prune.
2. **Single source of truth for cache paths.** The Spring configuration is the only place that names the cache
   directories; the JVM reads them from there. With a host cron, the path lived in both `application-prod.yml` and
   the crontab, and the two could drift.

#### Two-phase eviction

Each configured cache is processed independently and is walked exactly once. The walk materialises one entry per
regular file (path + size + last-access time) into an in-memory list; two filters then run against that list:

1. **Phase 1 — age-based prune (soft target).** Delete every regular file whose `atime` is older than
   `cleanup-max-age-days` (default 30 days). This is the day-to-day mechanism that keeps the cache lean.
2. **Phase 2 — size-based eviction (hard cap).** If the surviving total size still exceeds `maven-max-size` /
   `gradle-max-size` (defaults 3 GB / 6 GB), delete files in ascending order of `atime` (oldest used first) until
   the remaining total drops below `max-size × size-low-watermark-ratio` (default 0.75 → 2.25 GB / 4.5 GB).

After both phases, empty directories are swept post-order so the layout matches what Maven / Gradle would create
on the next miss. The hard cap dominates: if you set `cleanup-max-age-days` very high (or disable age-pruning
implicitly with a tiny watermark), the size cap still bounds total disk use.

> **Why `atime` and not `mtime`?** `mtime` records when an artifact was downloaded — it never changes once the
> file lands on disk, so an `mtime`-based prune re-downloads old-but-still-hot artifacts after every cycle.
> `atime` records when the file was last read, which is exactly what we want for an LRU cache. The default Linux
> mount option `relatime` bumps `atime` at most once per day (sufficient for our 30-day default) without the I/O
> overhead of strict `atime`. The cleanup task does not work correctly on filesystems mounted with `noatime`;
> use `relatime` or `strictatime` for the cache mount.

#### Pause semantics and race safety

The cleanup task takes an exclusive maintenance pause through the build agent's existing pause/resume mechanism:

- It calls `pauseForMaintenance()` on `SharedQueueProcessingService`. The call returns `true` only when this
  invocation actually transitioned the agent from running to paused.
- If the return is `false`, the agent was already paused (administrator pause, failure-backoff, or an overlapping
  cleanup cycle). The cleanup task **logs and aborts without resuming** — ownership of the pause stays with
  whoever already held it. This is enforced by an integration test
  (`testCacheCleanupDoesNotResumeAdminPause`).
- The pause causes the agent's listener and scheduled job-pickup task to be removed, so no new jobs are accepted.
  In-flight builds run to completion up to `pause-grace-period-seconds` (default 60 s); any straggler is force-
  cancelled and re-queued, where another live agent picks it up.
- The resume call is wrapped in a `finally`, so an exception inside the prune still resumes the agent.

The prune phase itself is bounded by a 15-minute wall-clock cap (checked at every loop iteration **and**
periodically inside the initial file walk so a hung NFS mount cannot block past the cap). If exceeded, the
in-flight phase finishes its current file, remaining files are skipped, and the agent is resumed. The next
scheduled invocation re-runs the cleanup from scratch. The cap is on the prune itself, not on the total
maintenance window: the drain of in-flight builds can additionally take up to `pause-grace-period-seconds`
(default 60 s) before the prune starts.

If the agent's pause is released while the prune is mid-way (an administrator clicks "Resume" or a remote
`resumeBuildAgentTopic` message fires), the cleanup detects this on the next loop iteration and aborts
immediately rather than continue deleting files behind the back of a build that is now allowed to run. The
abort is logged as `Pause was released during phase N (...) for <root>; aborting prune to avoid racing with
a live build.`

#### Per-agent staggering

`cleanup-cron` is a Spring 6-field cron expression and is templated per host by Ansible so multiple agents in the
same group do not drain simultaneously. The reference inventory uses a 3-minute offset based on the trailing
number in the agent's hostname:

```yaml
# group_vars/<agent group>.yml
continuous_integration:
  localci:
    build_container_cache:
      cleanup_cron: "0 {{ (inventory_hostname.split('.')[0] | regex_search('([0-9]+)$') | int - 1) * 3 }} 5 * * *"
```

That resolves to `0 0 5 * * *` on `agent01`, `0 3 5 * * *` on `agent02`, …, `0 57 5 * * *` on `agent20`. The
spread keeps total cluster capacity at ≥ (N − 1) / N at any moment during the maintenance window. 05:00 UTC was
chosen because it falls outside business hours for the European audience that Artemis is typically deployed for;
override per environment if your audience differs.

If two or more agents do happen to drain at the same time (e.g. multiple groups inheriting the same cron), the
cluster still degrades gracefully — pending builds wait briefly until one of them resumes — but the staggered
expression is preferred so the operator never has to wonder.

#### Read-only caches skip cleanup

When `read-only: true` is set, the bind mounts are read-only inside the container, so by definition the agent
cannot grow the cache and there is nothing for the cleanup task to prune. In that mode the cleanup short-circuits
with a log line (`Build-container cache is read-only; cleanup is the operator's responsibility, skipping.`) and
the agent is not paused. Operators using read-only caches own cache contents — typically a separate warm-up job
populates the cache out of band; the prune of that warm-up cache is on the operator's side.

#### Operator runbook

- **Disable cleanup temporarily** (e.g. while debugging) — set `cleanup-enabled: false` in
  `application-buildagent.yml` and restart the agent. The agent will keep serving builds; the cache will grow
  monotonically until you re-enable cleanup.
- **Force an immediate prune** — on the build-agent details page (`/admin/build-agents/details?agentName=…`),
  click **Run cache cleanup**. The agent flips to `MAINTENANCE` status while it pauses, prunes, and resumes;
  the operation is the same as the scheduled task.
- **Inspect what the prune did** — search the agent log for the line:
  `Cache prune for /var/cache/artemis-buildagent/m2: age-deleted=X files / X bytes; size-deleted=X files / X bytes;`
  Each invocation logs one such line per configured cache, plus the empty-directory sweep count and total
  duration.
- **Wipe the cache after suspected corruption** — on the build-agent details page, click **Reclaim disk…**, tick
  the Maven and/or Gradle box (the sum of files that will be deleted is shown next to each), type the confirmation
  word, and Confirm. The agent pauses, deletes every file under the chosen cache root regardless of age, then
  resumes. The next builds will re-download from Maven Central; the cache warms back up over the following hours.
- **Reclaim disk when Docker images dominate** — same Reclaim disk… dialog also offers **Unused Docker images**,
  which removes every image not bound to a running container regardless of `imageExpiryDays`. The next build that
  needs a freshly-removed image will pull it on assignment (slow); operators should only use this when an agent is
  actually close to full.
- **Adjust size caps for an agent with a tighter disk** — set `maven-max-size` / `gradle-max-size` in the agent
  group's inventory (Ansible `group_vars/<group>.yml`). Spring's `DataSize` parser accepts `MB`, `GB`, etc. The
  values render straight through to `application-prod.yml` on the host; no role change is needed.

The default values (3 GB Maven, 6 GB Gradle, 30-day age threshold, daily 05:00 UTC) are sized for the production
agents in the Artemis reference fleet (74 GB root filesystem, ~25–35 GB free). Tighten them for agents on smaller
disks; loosen for fat agents that serve many distinct exercises.

### Caveats

- **Cache poisoning between submissions (security trade-off).** When `read-only: false` is in effect, a student
  submission that successfully resolves a dependency writes the artifact into the shared host cache. Subsequent
  submissions — for the same student, a different student, or a different exercise on the same agent — will consume
  the cached copy without re-validating against Maven Central. A malicious build could deliberately seed the cache
  with a manipulated artifact under a coordinate that a future submission declares, which would then be silently
  consumed by that build. The pre-baked dependencies in the official Artemis build images (e.g.
  `ls1tum/artemis-maven-template`) cover the common case and are not affected because they are baked into the image
  layer, not the bind-mounted cache. **Mitigations, in order of strength:**
  1. Set `read-only: true` and populate the cache out of band via a trusted warm-up. Student submissions then
     cannot mutate the shared cache at all, so the entire class of cache-poisoning attacks goes away.
  2. Use the default `read-only: false` only on agents that run general programming exercises without secret
     tests, or in combination with the network-restriction patterns described under
     [Security Considerations](#security-considerations-for-secret-tests).
  3. For exam-grade isolation, the [Docker volumes approach below](#caching-docker-volumes) — which mounts the
     cache read-only for student submissions and read-write only for the trusted solution warm-up — gives the same
     guarantee with per-submission granularity, at the cost of operating an extra Jenkins-style pipeline layer.
- **Gradle file locks.** Gradle's `~/.gradle/caches` uses file locks; concurrent Gradle builds on the same agent will
  briefly serialize during dependency resolution. Maven has no equivalent lock. If contention becomes a hotspot with
  large `concurrent-build-size`, consider switching the Gradle cache to a read-only mount populated by a trusted
  warm-up build via Gradle's `GRADLE_RO_DEP_CACHE` environment variable (Gradle ≥ 6.1; configures a read-only
  fallback dependency cache that is not subject to file-lock contention) — out of scope for the default
  configuration.
- **First build per agent is still slow** because the cache is empty (cold-fill). Steady-state speedup applies from
  the second build onwards.


## Caching with Docker Volumes {/* #caching-docker-volumes */}

With Docker volumes you can cache Maven dependencies.
An alternative approach for caching is with Sonatype Nexus, see [this section](#caching-sonatype-nexus).


### Example for Maven

The container image used to run the maven-tests already contains a set of commonly used dependencies
(see [artemis-maven-docker](https://github.com/ls1intum/artemis-maven-docker)).
This significantly speeds up builds as the dependencies do not have to be downloaded every time a build is started.
However, the dependencies included in the container image might not match the dependencies required in your tests
(e.g. because you added new dependencies or the container image is outdated).

You can cache the maven-dependencies also on the machine that runs the builds
(that means, outside the container) by editing the `pipeline.groovy` template.

Adjust the `dockerFlags` variable:

```groovy
  dockerFlags = '#dockerArgs -v artemis_maven_cache:/maven_cache -e MAVEN_OPTS="-Dmaven.repo.local=/maven_cache/repository"'
```

Note that this might allow students to access shared resources (e.g. jars used by Maven), and they might be able
to overwrite them.
You can use [Ares](https://github.com/ls1intum/Ares) to prevent this by restricting the resources
the student's code can access.

Alternatively, you can restrict the access to the mounted volume by changing the `dockerFlags` to
```groovy
dockerFlags = '#dockerArgs -e MAVEN_OPTS="-Dmaven.repo.local=/maven_cache/repository"'
```
and changing the `testRunner` method into

```groovy
void testRunner() {
    setup()

    docker.image(dockerImage).inside(dockerFlags) { c ->
        runTestSteps()
    }
}

private void setup() {
    if (isSolutionBuild) {
        dockerFlags += " -v artemis_maven_cache:/maven_cache"
    } else {
        dockerFlags += " -v artemis_maven_cache:/maven_cache:ro"
    }
}
```

This mounts the cache as writeable only when executing the tests for the solution repository, and as read-only when
running the tests for students’ code.


### Example for Gradle

In case of always writeable caches you can set `-e GRADLE_USER_HOME=/gradle_cache` as part of the `dockerFlags`
instead of the `MAVEN_OPTS` like above.

For read-only caches like in the Maven example, define `setup()` as

```groovy
private void setup() {
    if (isSolutionBuild) {
        dockerFlags += ' -e GRADLE_USER_HOME="/gradle_cache"'
        dockerFlags += ' -v artemis_gradle_cache:/gradle_cache'
    } else {
        dockerFlags += ' -e GRADLE_RO_DEP_CACHE="/gradle_cache/caches/"'
        dockerFlags += ' -v artemis_gradle_cache:/gradle_cache:ro'
    }
}
```


### Security Considerations

When you are using secret tests as part of your exercise, you might want to disable network traffic leaving the CI run to avoid students leaking information.
Thanks to the fact that the cache is prepared while running for the solution, you can disable the network for students submissions.
Adjust `dockerFlags` and `mavenFlags` only for student submissions, like this:

```groovy
private void setup() {
    if (isSolutionBuild) {
        // handle docker flags
    } else {
        // handle docker flags
        // if not solution repo, disallow network access from containers
        dockerFlags += ' --network none'
        mavenFlags += ' --offline'
    }
}
```


## Security Considerations for Secret Tests

When you are using secret tests as part of your exercise, you might want to restrict network traffic leaving the CI run to avoid students leaking information.


### Jenkins

In Jenkins setups, you can restrict the network access by adjusting the `pipeline.groovy` script.
Add some flags to the `dockerFlags` variable:

```groovy
dockerFlags += '--add-host "artemis.example.com:10.0.73.42" \
    --network "artemis-restricted"'
```

Additionally, on the CI runner host you will have to create the `artemis-restricted` Docker network and some iptables firewall rules to restrict traffic:

```bash
docker network create --opt com.docker.network.bridge.name=artemis-restr artemis-restricted
iptables -I DOCKER-USER -i artemis-restr -j DROP
iptables -I DOCKER-USER -i artemis-restr -d $IP_OF_ARTEMIS_EXAMPLE_COM_CACHE -p tcp --dport 8443 -j ACCEPT
```


## Timeout Options

This setting is relevant only when using [Integrated Code Lifecycle Setup](/admin/production-setup/multiple-artemis-instances#integrated-code-lifecycle).

You can adjust possible [timeout options](/admin/production-setup/programming-exercise-adjustments#timeout-options) for the build process in Integrated Code Lifecycle Setup.
These values will determine what is the minimum, maximum, and default value for the build timeout in seconds that can be set in the Artemis UI.
The max value is the upper limit for the timeout, if the value is set higher than the max value, the max value will be used.

If you want to change these values, you need to change them in `localci` and `buildagent` nodes.
The corresponding configuration files are `application-localci.yml` and `application-buildagent.yml`.
Ensure that the values are consistent across these files. In a [multi-node setup](./multiple-artemis-instances.mdx), also verify that both core nodes and build agent nodes use the same values.

Default values have already been set, and modifying these values is not required unless a specific adjustment is needed.

```yaml
artemis:
    continuous-integration:
        build-timeout-seconds:
            min: <value>
            max: <value>
            default: <value>
```


## Build Log Configuration

This setting is relevant only when using [Integrated Code Lifecycle Setup](./integrated-code-lifecycle-setup.mdx).

The build log settings control the amount of log data stored per build job.
These values define the maximum number of lines and the maximum number of characters per line that can be retained in the logs.
The [Java Docker client](https://github.com/docker-java/docker-java) imposes a hard limit of 1024 characters per line.
If a lower value is configured, logs exceeding this limit will be truncated, and excess characters will be lost.
These settings only need to be adjusted in the buildagent configuration file, `application-buildagent.yml`.

Default values have already been set, and modifying these values is not required unless a specific adjustment is needed.

```yaml
artemis:
    continuous-integration:
        build-logs:
            max-lines-per-job: <value>
            max-chars-per-line: <value>
```


## Container Resource Restrictions

This setting is relevant only when using [Integrated Code Lifecycle Setup](./integrated-code-lifecycle-setup.mdx).

Instructors can edit the resources allocated to containers resulting from a programming exercise.
These restrictions ensure that assigned values do not exceed predefined limits.
To enforce these constraints, update the buildagent configuration files, `application-buildagent.yml`.
Ensure that these values remain consistent across all nodes in a [multi-node setup](./multiple-artemis-instances.mdx) to maintain uniform resource allocation.

Default values have already been set, and modifying these values is not required unless a specific adjustment is needed.

```yaml
artemis:
    continuous-integration:
        container-flags-limit:
            # Optional: restrict which custom Docker networks instructors can select
            # Comma-separated list. Use 'none' to allow "no network" option
            # This list must match networks available on the build agent host(s)
            # Example: none, artemis-restricted
            allowed-custom-networks: <network1>, <network2>
            max-cpu-count: <value>
            max-memory: <value>
            max-memory-swap: <value>
```

## Custom Docker Networks

This setting is relevant only when using [Integrated Code Lifecycle Setup](./integrated-code-lifecycle-setup.mdx).

Artemis allows instructors to select a custom Docker network for build containers per programming exercise. This can be used to
isolate containers (e.g., disable all outbound traffic) or to place them into a restricted network that only allows access to
approved services such as artifact caches.

- Configure the list of networks that instructors are allowed to choose from using the `allowed-custom-networks` property.
- Provide a comma-separated list of Docker network names. Use `none` to offer the "no network" option.
- Custom network names must exist on the build agent host(s) where containers are created (e.g., create with `docker network create ...`). The special value `none` is a built-in Docker option and does not need to be created.
- If no network is selected in the UI, Docker’s default network mode is used.

```yaml title="application-buildagent.yml"
artemis:
    continuous-integration:
        container-flags-limit:
            # Example: offer a fully isolated option and a restricted bridge network
            allowed-custom-networks: none, artemis-restricted
```

<Callout variant={CalloutVariant.info}>
    In distributed deployments, configure the same list on all core nodes. The build agent uses the network that the core node passes to it and expects the network to exist on the
    Docker host.
</Callout>

<Callout variant={CalloutVariant.warning}>
    Selecting `none` disables network access for the container. Ensure all dependencies are available in the image or are cached, and configure builds to run offline where
    applicable (e.g., Maven `--offline`), otherwise builds may fail.
</Callout>

## Pause Grace Period

This setting is relevant only when using [Integrated Code Lifecycle Setup](./integrated-code-lifecycle-setup.mdx).

The pause grace period determines how long the build agent waits after being paused before canceling all currently running jobs on that agent and adding them back to the queue.
This setting should be adjusted only if the default grace period is unsuitable for the specific environment. The configuration is set in the buildagent configuration file, `application-buildagent.yml`.

Default values have already been set, and modifying these values is not required unless a specific adjustment is needed.

```yaml
artemis:
  continuous-integration:
    pause-grace-period-seconds: <value>
```
